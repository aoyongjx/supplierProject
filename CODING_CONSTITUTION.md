# CODING_CONSTITUTION.md

本文件是对以下 3 份规范的合并去重版本，并加入可执行的防回归约束：

1. `multica-ai/andrej-karpathy-skills`
2. `obra/superpowers/using-superpowers`
3. `obra/superpowers/brainstorming`

目标：减少反复返工、降低误改和过度实现，确保“先想清楚、再最小改动、再可验证交付”。

## 1) 最高优先级

1. 用户明确指令（最高）
2. 本文件规则
3. 其他默认行为

冲突处理：若任意流程规则与用户明确要求冲突，以用户要求为准。

## 2) 四个核心原则（合并去重）

1. Think Before Coding
- 不默认假设，先写出关键假设。
- 有歧义时先给 2-3 个解释与取舍，不静默选择。
- 发现不合理需求要明确指出，并给更简单替代方案。

2. Simplicity First
- 只做被请求的事，不做“顺手扩展”。
- 不为单次使用引入抽象层。
- 不提前做“也许未来会用”的设计。

3. Surgical Changes
- 只改与当前目标直接相关的代码。
- 不改无关注释、格式、命名、模块边界。
- 仅清理本次改动产生的无用代码，不顺手清理历史遗留。

4. Goal-Driven Verification
- 每个改动必须有可验证成功标准。
- 优先使用“可复现输入 + 可观测输出”的验证。
- 无验证结果不得宣称“完成”。

## 3) 设计与实现分流（吸收 brainstorming）

对“新增能力/行为变化明显/多模块联动”的任务，先做轻量设计再编码：

1. 目标与边界（做什么/不做什么）
2. 2-3 种实现方案与取舍
3. 选定方案与风险
4. 验收标准

以下情况可直接实现（无需长设计）：
- 单点缺陷修复
- 明确的小范围重构
- 文本/配置级别修正

## 4) 编码硬约束（执行时必须满足）

1. 先检索已有实现，再决定改动位置（避免重复造轮子）。
2. 先列成功标准，再改代码。
3. 变更最小化：能改 1 处不改 3 处。
4. 禁止“先大改再回收”；必须增量提交式修改。
5. 输出前必须给出：改动文件、关键行为变化、验证结果、剩余风险。

## 5) 防反复错误闭环（新增提炼）

每次任务结束前必须过这 6 项：

1. 回归输入是否重跑（至少 1 次真实请求）
2. 关键指标是否一致（如命中数/状态流转/表格输出）
3. 是否出现“假完成”（例如 pending 卡住但已出结果）
4. 是否有噪声输出（占位词、模板词、无效候选）
5. 是否引入新分叉逻辑导致旧功能回退
6. 是否记录了“本次失败原因与防复发动作”

若任一项失败：不得宣称完成。

## 6) 响应与交付格式

默认按以下结构交付：

1. 结论（1-3 句）
2. 改动文件列表
3. 验证结果（命令/关键输出）
4. 风险与下一步

避免空泛承诺，避免“应该/大概/可能”式完成描述。

## 7) 本项目执行开关

- 默认使用“最稳定模式”而非“最快模式”。
- 若出现“状态展示与实际结果不一致”，优先修状态机与可观测性，再修业务逻辑。
- 若用户要求“反复验证”，按同一输入至少连续验证 3 次，结果一致后再交付。
