# 数据库设计

## 1. 文档定位

本文件基于当前后端实现（`server/index.js`）整理数据库设计，强调可审查性：

- 数据域划分
- 关键表与主外键关系
- 接口到表的映射关系
- 核心索引与查询场景
- 风险点与治理建议

## 2. Schema 与命名

- 业务 schema 通过环境变量 `DB_SCHEMA` 注入。
- 代码中通过 `const xxxTable = "${schemaName}"."table_name"` 统一引用。
- 服务启动时自动执行 `CREATE TABLE IF NOT EXISTS` 与关键索引初始化。

## 4. 核心关系模型

## 4.1 任务关系

- `crawl_task` 1:N `crawl_task_log`（`task_id`）
- `crawl_task` 1:1 `crawl_task_quality`（`task_id` 唯一）
- `crawl_info` 与供应链节点、供应商记录通过业务字段关联

## 4.2 供应链关系

- `supply_chain_node.parent_id -> supply_chain_node.id`（自关联树）
- `gas_supply_chain_node.parent_id -> gas_supply_chain_node.id`
- `supplier_base_info.node_id -> supply_chain_node.id`
- `gas_supplier.gas_node_id -> gas_supply_chain_node.id`

## 4.3 档案关系

- `supplier_profile` 1:N 多子项表，子项通过 `profile_id` 关联主档案。

## 4.4 知识库关系

- `knowledge_base` 1:N `knowledge_base_document`（`kb_id`）
- `knowledge_base_document` 1:N `knowledge_base_vector`（`doc_id`）

## 4.5 Wiki 图谱关系

- `llm_wiki_graph_edge` 关联源节点与目标节点（`target_id` + 关系类型）
- `llm_wiki_sync_task` 记录同步任务状态，支撑可追溯性

## 6. 索引设计（按场景）

### 6.1 列表分页与排序

- `idx_asset_inventory_updated_at`
- `idx_crawl_task_updated_at`
- `idx_chat_session_updated_at`
- `idx_model_provider_updated_at`

用途：按更新时间倒序分页的高频列表。

### 6.2 状态与过滤

- `idx_asset_inventory_status`
- `idx_crawl_task_status`
- `idx_kb_doc_status`

用途：状态筛选、待处理/失败数据快速定位。

### 6.3 关联查询

- `idx_crawl_log_task_id`
- `idx_kb_doc_kb_id`
- `idx_kb_vector_doc_id`

用途：父子表聚合展示与详情展开。

### 6.4 树结构与来源追溯

- `idx_supply_chain_node_parent`
- `idx_gas_supply_chain_node_parent`
- `idx_supply_chain_node_source`

用途：树展开、来源链接与实体回溯。

### 6.5 文本与向量相关

- `idx_supplier_opinion_supplier_name`

### 6.6 Wiki 同步与图谱

## 7. 关键字段建议（现状约束）

1. ID 策略
- 主数据与任务表混合使用 `BIGINT`/`VARCHAR` 主键，接口层需保持类型稳定。

2. 时间字段
- 大多数表有 `created_at/updated_at`，应统一使用 UTC 存储与前端时区展示转换。

3. 状态字段
- 任务、文档、同步表均存在状态字段，建议持续维护状态机枚举一致性。

4. 溯源字段
- 采集类表普遍有来源 URL/文件/任务 ID，需在导入链路保留。

## 8. 风险点与治理建议

1. 单库高耦合风险
- 多业务域共享同库，后续可按域拆 schema 或引入归档策略。

2. 任务数据增长风险
- `crawl_task_log`、`precise_sourcing_run_log` 体量增长快，建议增加归档/TTL 策略。

3. 文档与向量存储成本
- `knowledge_base_vector` 与 `supplier_opinion_vector` 增长显著，建议分层存储或冷热分离。

4. 索引膨胀风险
- 需定期审查低命中索引，避免写放大影响。

5. 结构变更风险
- 当前启动自动建表适合快速迭代，生产建议补充迁移脚本与版本化 DDL。
