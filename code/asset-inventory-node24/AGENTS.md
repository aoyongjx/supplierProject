## Execution Mode

- `fast mode`：严格关闭。
- 禁止以速度优先替代正确性，不允许跳过必要检索、差异检查、验证和关键上下文确认。
- 默认采用“先检索、后实现、再验证”的慢思考流程；遇到多阶段任务时，不因为赶进度省略中间校验。

## Strict Context Compression

- 严格执行上下文压缩。
- 在以下时机必须主动压缩上下文：仓库检索完成后、方案确定后、批量改文件后、验证完成后、以及出现大量日志/临时输出后。
- 压缩后只保留稳定事实，删除重复描述、临时推测、整段日志、无关命令输出和已失效分支。

## Compression Output Format

每次上下文压缩后的保留内容必须限制为以下 6 类：

1. 当前目标
2. 已确认约束
3. 已做决策
4. 已修改文件
5. 未解决风险/阻塞
6. 下一步动作

- 若历史上下文已经变长或出现分叉，继续工作前先用不超过 6 条短句重述上述内容。
- 未经确认的猜测不得写入压缩结果。
- 敏感信息、密钥、令牌、cookie、临时凭据不得写入任何压缩摘要或项目记忆。

## Change Discipline

- 改动保持最小、可读、可回滚。
- 优先复用现有实现，避免重复造轮子。
- 不得因为“快速修复”引入隐式行为变化。

## Verification

- 完成代码或配置改动后，至少执行与改动直接相关的验证。
- 最低要求：检查 diff，并运行必要的 `build`、`lint`、`test` 或针对性命令。
- 无法验证时，必须明确说明阻塞原因与当前风险。
